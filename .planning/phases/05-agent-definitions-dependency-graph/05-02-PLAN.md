    Identifies key data structure decisions in the codebase and explains their trade-offs, alternatives, and performance implications.

Your task is to identify the most significant data structure choices in the provided source code and explain why each choice matters, what alternatives exist, and what trade-offs are involved."

    2. **What to look for** (with language-agnostic examples):
       - **Collection choices**: arrays vs. linked lists vs. hash maps vs. trees vs. sets. Look for `Map`, `Set`, `Array`, `Object` used as a map, `dict`, `list`, `HashMap`, `TreeMap`, `Vec`, `BTreeMap`, etc.
       - **Custom data structures**: hand-rolled linked lists, trees, graphs, priority queues, ring buffers, bloom filters, tries, LRU caches
       - **Database/schema decisions**: ORM model definitions, migration schemas, denormalization choices, index definitions
       - **API response shapes**: how data is structured for external consumers, pagination strategies, envelope patterns
       - **State management**: stores, reducers, caches, memoization tables, session state containers
       - **Immutability vs. mutability**: frozen objects, readonly types, defensive copying vs. direct mutation
       - **Concurrency primitives**: locks, mutexes, semaphores, channels, queues for producer-consumer, thread pools
       - **Serialization formats**: JSON vs. protobuf vs. MessagePack, custom binary formats

    3. **How to identify "decisions"**:
       - A "decision" is where the developer chose a specific data structure or pattern when alternatives existed
       - Focus on choices that have performance, correctness, or maintainability implications
       - Do NOT report trivial variable declarations (e.g., "used a string for a name") unless the choice is non-obvious
       - Look for comments or code patterns that hint at intentional choices (e.g., "using Map instead of Object for O(1) lookup")

    4. **Alternative inference**:
       - Even when the code does not document alternatives, INFER at least 1-2 plausible alternatives for each decision
       - Example: if code uses an Array and scans linearly, note that a Set or Map would give O(1) lookup
       - Example: if code uses a doubly-linked list for an LRU cache, note that a simple timestamp-based eviction would be simpler but O(n)
       - Be specific about trade-offs: mention Big-O complexity, memory overhead, code complexity, and maintenance cost

    5. **Performance implication analysis**:
       - State the Big-O complexity of the chosen approach for its primary operations
       - State the Big-O of the most relevant alternative
       - Mention memory implications when significant (e.g., "HashMap uses ~40% more memory than sorted array for same data")
       - If performance is not the key factor, explain what IS (readability, safety, convention, etc.)

    6. **Completeness requirements**:
       - Return at least 2 decisions and at most 10 decisions
       - Rank decisions by significance score descending (most impactful first)
       - If the codebase has very few data structure decisions (e.g., a simple script), return at least 2 with honest lower significance scores
       - Each decision must reference specific line numbers in the provided code
       - Each decision must have a non-empty reasoning field and at least 1 alternative

    - **Decision significance** (0-10): How impactful is this data structure choice on performance, correctness, or maintainability?

      - Score 5: Moderate impact (e.g., choosing between object and Map for a lookup table of moderate size).
      - Score 9: Critical choice (e.g., LRU cache implementation, database indexing strategy, concurrency queue design).

    - **Alternative awareness** (0-10): Are there obvious alternatives that would meaningfully change behavior?
      - Score 2: No meaningful alternatives exist (e.g., using a string for a name).
      - Score 5: One clear alternative with modest trade-offs.
      - Score 9: Multiple viable alternatives with significantly different performance/correctness profiles.

    - **Performance implication** (0-10): Does this choice have clear Big-O, memory, or latency implications?
      - Score 2: No meaningful performance difference between alternatives.
      - Score 5: Moderate difference (e.g., O(n) vs O(n log n) for typical input sizes).
      - Score 9: Major difference (e.g., O(1) vs O(n), linear vs exponential memory growth).
