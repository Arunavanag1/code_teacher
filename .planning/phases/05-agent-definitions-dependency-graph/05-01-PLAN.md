    Replace the entire contents of agents/definitions/dependency-mapper.md with a production-quality agent definition. The file must have these exact section headers: # Dependency Mapper, ## Role, ## System Prompt, ## Input, ## Scoring Rubric, ## Output Schema.

    **## Role** (keep concise, 1-2 sentences):
    Maps imports, exports, function calls, class inheritance, and cross-file dependencies to build a scored dependency graph of the project.

    **## System Prompt** (multi-paragraph, detailed):
    Write the system prompt as a precise set of instructions covering:

    1. **Identity and task statement**: "You are a dependency analysis agent. Your task is to analyze the provided source code files and identify ALL cross-file dependencies, producing a structured dependency graph with per-node scoring."

    2. **What to look for** (language-agnostic examples for each pattern):
       - Import/export statements: ES modules (`import`/`export`), CommonJS (`require`/`module.exports`), Python (`import`/`from X import Y`), Go (`import`), Java (`import`), Rust (`use`/`mod`), C/C++ (`#include`)
       - Function/method calls across file boundaries: trace where exported functions are invoked by other files
       - Class inheritance: `extends`, `implements`, Python `class Foo(Bar)`, Go embedding
       - Interface implementations: TypeScript `implements`, Go implicit interfaces, Java `implements`
       - Shared state: singletons, global variables, shared configuration modules
       - Event systems: pub-sub patterns, event emitters/listeners, callback registrations
       - Re-exports: barrel files (index.ts/index.js) that re-export from other modules

    3. **Scoring instructions**:
       - Fan-in: count how many other files import or depend on this file. Score 0 if none, 1-3 for 1-2 dependents, 4-6 for 3-5, 7-9 for 6-10, 10 for 11+.
       - Fan-out: count how many other files this file imports from. Same scale.
       - Coupling depth: how deep in the import chain is this file? Entry points are 0. Files imported only by entry points are 1. Score is normalized to 0-10 based on max depth in the project.
       - Centrality: files that are both highly imported AND import many others are central hubs. Score = weighted combination of fan-in and fan-out. A pure leaf (high fan-in, zero fan-out) scores ~6. A hub (high fan-in AND high fan-out) scores 8-10.

    4. **Edge type classification**:
       - "imports": standard import/require/include dependency
       - "calls": function call across file boundary (not just importing the module, but invoking a specific function)
       - "extends": class inheritance
       - "implements": interface implementation
       - "uses": indirect dependency (event listener, configuration reference, shared state)
       - Weight (1-10): 10 = tight coupling (extends/implements), 7-9 = direct function call, 4-6 = standard import, 1-3 = indirect/loose coupling

    5. **Completeness requirements**:
       - Include EVERY file visible in the provided context as a node, even files with zero dependencies
       - Return at least 3 nodes even for very small projects
       - If a file has no imports and no dependents, still include it as an isolated node with all scores at 0
       - Do NOT hallucinate dependencies that are not visible in the provided code
       - Use relative file paths (relative to project root) as node IDs

    6. **Node type classification**:
       - "file": the default, when the node represents a whole file
       - "module": when the file is a barrel/index file that re-exports
       - "class": when a file's primary export is a single class
       - "function": when a file's primary export is a single function

    **## Input** (first sentence is the task instruction extracted by buildTaskInstruction):
    Analyze the provided source files and build a complete dependency graph with per-node fan-in, fan-out, coupling depth, and centrality scores. You will receive the full file contents and the project structure tree. Trace every import, export, class inheritance, and cross-file function call visible in the code.

    **## Scoring Rubric**:
    Criteria (each 0-10):
    - **Fan-out** (0-10): How many other modules does this file depend on? 0 = no dependencies, 5 = moderate (3-5 imports), 10 = heavy (10+ imports).
    - **Fan-in** (0-10): How many other modules depend on this file? 0 = no dependents, 5 = moderate (3-5 dependents), 10 = critical hub (10+ dependents).
    - **Coupling depth** (0-10): How deep in the dependency chain? 0 = entry point/leaf, 5 = mid-level module, 10 = deeply nested in longest chain.
    - **Centrality** (0-10): Is this a hub in the dependency graph? 0 = isolated node, 5 = moderate connectivity, 10 = central hub with high fan-in AND fan-out.

    Example scores for reference:
    - An entry point file (e.g., index.ts) that imports 8 modules but nothing imports it: fanOut=8, fanIn=0, couplingDepth=0, centrality=3
    - A utility module (e.g., helpers.ts) imported by 12 files with 2 own imports: fanOut=2, fanIn=10, couplingDepth=2, centrality=7
    - A core engine file imported by 6 files that itself imports 5: fanOut=5, fanIn=7, couplingDepth=3, centrality=9
    - An isolated test helper with no imports or dependents: fanOut=0, fanIn=0, couplingDepth=0, centrality=0

    **## Output Schema**:
    Expected JSON output format:
    ```json
    {
      "nodes": [
        {
          "id": "src/core/engine.ts",
          "type": "file",
          "exportedSymbols": ["Engine", "run", "stop"],
          "fanIn": 8,
          "fanOut": 3,
          "couplingDepth": 2,
          "centrality": 9
        },
        {
          "id": "src/utils/helpers.ts",
          "type": "file",
          "exportedSymbols": ["formatDate", "parseConfig"],
          "fanIn": 12,
          "fanOut": 1,
          "couplingDepth": 3,
          "centrality": 7
        }
      ],
      "edges": [
        {
          "source": "src/index.ts",
          "target": "src/core/engine.ts",
          "type": "imports",
          "weight": 5
        },
        {
          "source": "src/core/engine.ts",
          "target": "src/utils/helpers.ts",
          "type": "calls",
          "weight": 7
        }
      ],
      "summary": "The project has a hub-and-spoke architecture centered around src/core/engine.ts which orchestrates 8 downstream modules. The utils/ directory serves as a shared foundation with high fan-in."
    }
    ```

    IMPORTANT: The output file must contain literal triple-backtick fences in the Output Schema section.
    Ensure the complete file content is written, not a partial update.

    **## Role** (keep concise, 1-2 sentences):
    Scores code sections for educational value -- identifying patterns, algorithms, and implementations most valuable for teaching and learning.

    **## System Prompt** (multi-paragraph, detailed):
    Write the system prompt covering:

    1. **Identity and task statement**: "You are a code education analysis agent. Your task is to evaluate the provided source code and identify sections that are most valuable for teaching and learning. For each notable section, produce a detailed score across five criteria with plain-English explanations."

    2. **What makes code teachable** (with language-agnostic examples):
       - Algorithmic implementations: sorting, searching, graph traversal, dynamic programming, sliding window, two pointers, BFS/DFS, divide and conquer
       - Design pattern implementations: factory, observer, strategy, decorator, middleware, pub-sub, singleton (when well-done), builder
       - Data structure implementations or usage: custom linked lists, trees, hash maps, priority queues, LRU caches, ring buffers, tries
       - Error handling patterns: try/catch hierarchies, Result/Either types, error boundary patterns, retry logic with backoff
       - Clean code examples: clear naming, separation of concerns, single responsibility, well-structured abstractions
       - Concurrency patterns: async/await orchestration, worker pools, mutex patterns, channels, producer-consumer
       - Progressive complexity: code that builds on simpler concepts in a natural learning progression

    3. **Section identification rules**:
       - A "section" is a contiguous range of lines in a single file (startLine to endLine)
       - Sections should be logical units: a complete function, a class, a module's core logic, a pipeline
       - Minimum section size: 5 lines. Maximum: 150 lines. If a pattern spans more, pick the most illustrative subsection
       - Sections must reference actual line numbers from the provided file content
       - Do NOT identify boilerplate, auto-generated code, simple getters/setters, or trivial type definitions as teachable

    4. **Scoring calibration** (anchored to concrete definitions):
       - A score of 1-2 means: "Barely present, trivial, nothing interesting to learn here"
       - A score of 4-5 means: "Standard implementation, useful but not remarkable"
       - A score of 7-8 means: "Well-done, demonstrates clear expertise and good practices"
       - A score of 9-10 means: "Exemplary, best-practice reference material, would use in a textbook"
       - The overall score is the average of all 5 criterion scores

    5. **Completeness requirements**:
       - Return at least 3 sections and at most 15 sections
       - Rank sections by overall score descending (highest score first)
       - If the codebase has fewer than 3 clearly teachable sections, include the 3 best candidates anyway with honest lower scores
       - Every section must have a reasoning field explaining what makes it educational in plain English
       - Every section must list concepts (what can be learned) and prerequisites (what you need to know first)

    6. **Prerequisite inference**:
       - Prerequisites are concepts a learner must understand BEFORE studying this section
       - Examples: "async/await", "hash maps", "Big-O notation", "HTTP basics", "recursion", "type generics"
       - List 1-5 prerequisites per section, from most to least important
       - If a section is self-contained (understandable without prerequisites), use an empty array

    **## Input** (first sentence is the task instruction):
    Analyze the provided source files and identify the most teachable code sections, scoring each on a 0-10 scale across five criteria. You will receive the full file contents and the project structure tree. Focus on sections that demonstrate patterns, algorithms, data structures, or design decisions in a clear and transferable way.

    **## Scoring Rubric**:
    Criteria (each 0-10):
    - **Conceptual density** (0-10): How many distinct learnable concepts appear in this section? A simple getter returns 1. A function using recursion + memoization + hash maps scores 7+. A section combining multiple design patterns scores 9+.
      - Score 2: Single basic concept (a loop, a conditional).
      - Score 5: Two or three concepts combined (e.g., async function with error handling).
      - Score 9: Four or more concepts woven together (e.g., observer pattern + event loop + error boundaries + clean separation).

    - **Clarity** (0-10): How readable and understandable is the implementation? Consider naming, structure, comments, and code organization.
      - Score 2: Cryptic variable names, dense one-liners, no comments on complex logic.
      - Score 5: Reasonable names, standard structure, could be followed with some effort.
      - Score 9: Self-documenting code, excellent naming, logical flow, comments that explain "why" not "what".

    - **Transferability** (0-10): Are the patterns and techniques applicable beyond this specific project?
      - Score 2: Highly project-specific logic (e.g., parsing a proprietary format).
      - Score 5: Common pattern implemented in a standard way (e.g., CRUD operations).
      - Score 9: Universal technique applicable across domains (e.g., rate limiting, caching strategy, retry logic).

    - **Novelty** (0-10): Does this show something a learner likely has not seen, or implements a common thing uncommonly well?
      - Score 2: Textbook boilerplate (e.g., basic REST endpoint).
      - Score 5: Standard pattern with a noteworthy twist or optimization.
      - Score 9: Unusual approach, clever optimization, or elegant solution to a hard problem.

    - **Self-containment** (0-10): Can this section be understood without reading the rest of the project?
      - Score 2: Requires deep understanding of the project's architecture, types, and conventions.
      - Score 5: Needs some context (a few types or utilities from elsewhere).
      - Score 9: Fully self-contained, could be extracted and understood in isolation.

    **## Output Schema**:
    Expected JSON output format:
    ```json
    {
      "sections": [
        {
          "file": "src/algo/rate-limiter.ts",
          "startLine": 10,
          "endLine": 67,
          "score": 9.4,
          "criteria": {
            "conceptualDensity": 9,
            "clarity": 10,
            "transferability": 10,
            "novelty": 8,
            "selfContainment": 10
          },
          "reasoning": "Sliding window rate limiter using a sorted set. Demonstrates time-based data expiration, O(log n) insertion, and clean API design. The implementation is self-contained and the pattern applies to any rate-limited system.",
          "concepts": ["sliding window", "sorted sets", "rate limiting", "time complexity"],
          "prerequisites": ["hash maps", "Big-O notation", "timestamps"]
        },
        {
          "file": "src/middleware/auth.ts",
          "startLine": 30,

          },
          "reasoning": "JWT validation middleware demonstrating the decorator pattern, clean error propagation, and middleware chaining. Well-structured with clear separation between token verification and authorization logic.",
          "concepts": ["middleware pattern", "JWT validation", "decorator pattern", "error propagation"],
          "prerequisites": ["HTTP basics", "async/await", "JSON Web Tokens"]
        }
      ]
    }
    ```
